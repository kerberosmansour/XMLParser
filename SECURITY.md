# Security Rules For XMLParser

> Generated by `/slo-architect` on 2026-05-09.
> Every downstream agent should treat this file as the source of truth for security defaults in this project.

## Stack

- Primary stack: C++17 library + CMake.
- Runtime: C++17 standard library only.
- Test-time tools: Catch2, compiler sanitizers, coverage tooling, static analysis.

## Top Risks

~~~text
- Breach: Callers may parse XML containing secrets or personal data; the library must never log raw XML payloads, entity replacement text, or internal file paths by default.
- Compliance fine: Consumers may use the library in regulated products; incorrect entity handling or malformed-input crashes could undermine OWASP ASVS-style input validation obligations in downstream systems.
- Prolonged outage: A malicious or accidental oversized XML document, excessive nesting, or entity expansion could exhaust CPU or memory in a service using the parser.
~~~

## Input Handling

- Treat all XML bytes, streams, DTD declarations, entity values, namespace names, and processing instruction contents as untrusted input.
- Validate encoding before structural parsing.
- Enforce configured bounds for document bytes, element depth, token length, attribute count, entity expansion, DTD declaration count, and DOM node count.
- Reject malformed, truncated, empty, or null-equivalent input with a typed error. Never rely on assertions for public input validation.
- Optional external DTD fetching is disabled by default. If enabled, it must use a caller-provided resolver with explicit allow-lists and bounded time/size behavior.

## Error Handling

- Well-formedness errors and validity errors must be distinguishable.
- Public parse errors include message, line, column, and byte offset.
- Error messages must not include raw XML payload snippets by default.
- SAX recoverable error callbacks must not suppress resource-limit or well-formedness failures.
- Exceptions thrown by caller callbacks must not leak resources or corrupt parser/DOM state.

## Memory And Resource Safety

- Use RAII throughout.
- Public API ownership uses standard library types; no raw owning pointers in public interfaces.
- Internal stacks and buffers must have explicit expected and hard bounds where growth is attacker-controllable.
- Parser recursion must be iterative or depth-checked before recursive descent.
- Fuzz/sanitizer failures are security-relevant until proven otherwise.

## Supply Chain

- Runtime dependencies: none beyond the C++ standard library.
- Test-only dependency licenses must be reviewed and documented before pinning.
- CMake dependency downloads, if used for tests, must be pinned. CI should prefer reproducible package restore or vendored test framework sources.
- Secret scans and dependency audits are required in CI once tooling exists.

## Compliance

Default columns for threat modeling: SOC 2 and OWASP ASVS. This project is a parsing library, not an application data processor, so GDPR/HIPAA/PCI controls are downstream consumer responsibilities unless a future feature stores or transmits caller XML.

## Out Of Scope

- Authentication and authorization: not applicable to this library.
- Transport TLS: not applicable unless optional external DTD fetching is implemented by this library rather than by a caller resolver.
- Incident response: downstream application responsibility.

# Threat Model - XMLParser

> Generated by `/slo-architect` on 2026-05-09 and strengthened during design review.
> Scope: components and surfaces described in [ARCHITECTURE.md](../../../ARCHITECTURE.md).
> This file is cited by `/slo-plan`, `/slo-critique`, `/slo-execute`, and `/slo-verify`.

## System Description

~~~text
XMLParser is a C++17 library that accepts caller-provided XML bytes or chunks, detects supported encodings, parses XML structure, resolves namespaces, optionally validates internal DTD declarations, emits SAX events, builds DOM trees, and serializes DOM trees back to XML. The library is embedded inside caller applications and therefore treats XML input, DTD content, namespace names, entity values, processing instruction contents, and output streams as untrusted boundaries.
~~~

## Top Risks

~~~text
- Breach: Callers may parse XML containing secrets or personal data; the library must never log raw XML payloads, entity replacement text, or internal file paths by default.
- Compliance fine: Consumers may use the library in regulated products; incorrect entity handling or malformed-input crashes could undermine OWASP ASVS-style input validation obligations in downstream systems.
- Prolonged outage: A malicious or accidental oversized XML document, excessive nesting, or entity expansion could exhaust CPU or memory in a service using the parser.
~~~

## Security Objectives

| Objective | Why It Matters | Requirement Link |
|---|---|---|
| Parse untrusted XML without undefined behavior | Malformed XML is the normal adversarial input for a parser. | REQ-ERR-06 |
| Preserve source-location accuracy | Callers need actionable diagnostics without raw payload leakage. | REQ-ERR-02 |
| Bound attacker-controlled growth | XML nesting, attributes, entities, and DOM nodes can exhaust resources. | REQ-ERR-05, REQ-ERR-06 |
| Keep DOM and SAX behavior consistent | Divergent parsers create conformance and security gaps. | REQ-DOM-01, REQ-SAX-01 |
| Disable external DTD access by default | External entity fetching can become SSRF or local file disclosure in consumers. | REQ-ERR-04 |
| Keep public API dependency-light and portable | Consumers need predictable embedding across platforms. | REQ-PLAT-01 through REQ-PLAT-06, REQ-API-02 |

## Assets And Data Classification

| Asset | Classification | Protection Need |
|---|---|---|
| Caller-provided XML bytes/chunks | Confidential in many downstream applications | Never log by default; never leak snippets in diagnostics unless caller opts in |
| Entity replacement text and DTD declarations | Confidential/Internal | Bound expansion; avoid raw values in errors |
| Source locations | Internal | Safe to expose because they are structural metadata |
| DOM tree contents | Confidential in many downstream applications | Owned by caller; no hidden persistence or telemetry |
| Parser options and limits | Public/Internal | Validate and keep finite defaults |
| Conformance fixtures | Public/Internal | Track source and license; do not mix fixture content with implementation provenance |
| Build/package metadata | Public | Keep deterministic and consumer-safe |

## Actors

| Actor | Trust Level | Capabilities |
|---|---|---|
| Library consumer developer | Trusted to call APIs, but may misuse them | Configures parser options, DOM mutation, SAX callbacks, external DTD resolver |
| XML document supplier | Untrusted | Controls bytes, encoding declarations, markup, namespaces, DTD/internal subset, entity references, PIs, comments |
| Network peer feeding chunks | Untrusted | Controls chunk boundaries and timing before the caller forwards buffers |
| SAX callback code | Caller-owned, partially trusted | Runs during parse and may throw or mutate external state |
| Output stream implementation | Caller-owned, partially trusted | May fail or throw during serialization |
| Build/test fixture source | External, partially trusted | Provides conformance data and dependency metadata |

## Trust Boundaries And Entry Points

| Boundary / Entry Point | Trust Transition | Primary Controls |
|---|---|---|
| `parse(std::string_view)` / stream parse | untrusted XML into library | encoding validation, grammar checks, bounded limits, typed errors |
| `SaxParser::feed(std::string_view)` | arbitrary chunk boundaries into parser state | carryover invariants, finish-state validation, buffer caps |
| SAX callbacks | library into caller code | RAII, exception propagation, no internal state corruption |
| DOM mutation APIs | caller requests tree mutation | ownership checks, parent/child invariants, no cycles |
| Serialization to `std::ostream` | library into caller output sink | stream failure checks, typed serialization failure |
| Internal DTD subset | untrusted declarations into validation tables | declaration validation, duplicate checks, entity expansion limits |
| Optional external DTD resolver | library may cross filesystem/network through caller code | disabled by default, caller resolver allow-list contract, byte/time limits |
| CMake package/test fixture intake | external metadata into repo/build | pinned test dependency, fixture license manifest, install-tree consumer tests |

## Assumptions

- The library has no authentication, authorization, network listener, database, or durable storage surface.
- The library may be embedded in security-sensitive products, so diagnostics must be safe by default.
- XML Schema, XPath, XQuery, XSLT, and binary XML remain out of scope for this runbook.
- External DTD fetching is not a built-in default behavior; any future resolver is caller-provided and opt-in.
- C++ exceptions are allowed by the public error model; exception safety is therefore a design requirement, not an implementation nice-to-have.

## Abuse Cases Per Surface

| Surface | Attacker | Attack Step | Desired Outcome | Control | Threat-Model Row ID |
|---|---|---|---|---|---|
| Public parse input | Malicious document supplier | Provide deeply nested elements and huge attribute lists | Exhaust stack or heap in a consuming service | `ParserOptions` depth, attribute, node, and byte limits | tm-xml-parsing-library-abuse-1 |
| Public parse input | Malicious document supplier | Provide malformed UTF-16 or truncated multibyte input | Trigger undefined behavior or incorrect source offsets | strict decoder with byte-offset tests and typed encoding errors | tm-xml-parsing-library-abuse-2 |
| Public parse input | Malicious document supplier | Provide a payload whose raw text looks like secrets | Make diagnostics leak sensitive data to logs | safe error messages that omit raw XML by default | tm-xml-parsing-library-abuse-3 |
| Incremental SAX feed | Network peer | Split multibyte characters across arbitrary buffers | Desynchronize decoder or corrupt byte offsets | carryover buffer invariants and chunk-boundary tests | tm-xml-parsing-library-abuse-4 |
| Incremental SAX feed | Network peer | End stream in the middle of markup or CDATA | Parser accepts truncated document | `finish()` state validation with typed truncated-input error | tm-xml-parsing-library-abuse-5 |
| Incremental SAX feed | Network peer | Feed many tiny chunks | Force quadratic buffering or excessive reallocations | bounded carryover and amortized buffer tests | tm-xml-parsing-library-abuse-6 |
| SAX callback boundary | Application callback author | Throw during nested element callbacks | Leak resources or corrupt parser state | RAII and exception propagation tests | tm-xml-parsing-library-abuse-7 |
| SAX callback boundary | Application callback author | Re-enter parser from a callback if API permits | Corrupt parser state machine | re-entrancy guard or documented non-reentrant contract with tests | tm-xml-parsing-library-abuse-8 |

| Serializer output | Malicious DOM builder path | Serialize text containing `<`, `&`, or invalid chars | Emit malformed XML or inject markup | context-aware XML escaping and char validation | tm-xml-parsing-library-abuse-14 |

| Internal DTD/entity layer | Malicious document supplier | Provide recursive or expanding entity definitions | Exhaust CPU or memory | entity recursion, expansion count, and replacement length limits | tm-xml-parsing-library-abuse-16 |
| Internal DTD/entity layer | Malicious document supplier | Define duplicate or conflicting declarations | Bypass validity checks or confuse errors | declaration table duplicate checks and validity diagnostics | tm-xml-parsing-library-abuse-17 |
| Internal DTD/entity layer | Malicious document supplier | Reference undeclared entities in validating mode | Parser treats invalid content as valid | validity error separation and recoverable callback tests | tm-xml-parsing-library-abuse-18 |
| Optional external DTD resolver | Unsafe integrator | Resolve `file://` or private network URLs | SSRF or local file disclosure in consumer application | feature off by default; resolver allow-list contract | tm-xml-parsing-library-abuse-19 |
| Optional external DTD resolver | Malicious document supplier | Point resolver at a very large external subset | Memory/time exhaustion | resolver byte and timeout limits | tm-xml-parsing-library-abuse-20 |
| Optional external DTD resolver | Malicious document supplier | Cause resolver failure or slow response | Hang parse or hide validation result | bounded resolver failure handling and explicit diagnostics | tm-xml-parsing-library-abuse-21 |

## Milestone Security Gates

| Milestone | Gate |
|---|---|
| M1 | Public API skeleton must expose finite defaults and safe placeholder errors; no raw XML in diagnostics. |
| M2 | Encoding/tokenizer must reject malformed/truncated input with correct source locations and no UB. |
| M3 | SAX incremental parser must survive arbitrary chunk boundaries and callback exceptions without leaks/corruption. |
| M4 | DOM mutation must prevent ownership cycles, duplicate ownership, and namespace-invalid serialization. |
| M5 | DTD validation must separate validity from well-formedness and prove external DTD resolution remains opt-in. |

## Residual Risks

| Risk | Exploit Path | Compensating Control | Owner | Review By |
|---|---|---|---|---|
| XML conformance gaps before fixture harness is complete | Unsupported edge cases may parse incorrectly | Milestone 1 conformance matrix blocks parser completion claims | Maintainer | Before M2 starts |
| Optional external DTD resolver misuse | Caller resolver fetches unsafe resources | Feature disabled by default; resolver contract and tests required before enablement | Maintainer and integrator | M5 |
| Resource limit defaults may be too strict or too loose | Real workloads fail or malicious inputs consume too much | Defaults documented, configurable, and stress-tested | Maintainer | M5 release gate |
| Exception-based API may not fit all consumers | Consumers in no-exception builds cannot use primary API | Consider non-throwing adapters after v1 core is stable | Maintainer | Post-v1 backlog |
| Full XML 1.1 edge cases may lag XML 1.0 | XML 1.1 is less commonly exercised and can regress | Dedicated XML 1.1 conformance subset and version-specific tests | Maintainer | M5 |

## Review Checklist

- Every public input boundary has validation and a typed failure mode.
- Every untrusted growth path has a declared bound and a test.
- Every callback or stream boundary has exception/failure tests.
- Every diagnostic path avoids raw XML payload disclosure by default.
- Every requirement category has at least one planned evidence artifact.
- Every optional external interaction is disabled by default and requires caller opt-in.
